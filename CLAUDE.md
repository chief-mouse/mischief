# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Mischief (`mschf`) is a BeeWare/Briefcase desktop app (Toga UI) that acts as a "Workspace Manager" for micro-apps. A micro-app is a single `.msf` file — a SQLite database containing app metadata, dill-pickled Python UI code, RBAC rules, and a ledger of cryptographically signed transactions. The host loads an `.msf`, verifies signatures against a local X.509 CA, enforces RBAC, and executes the app's code in a sandbox to render native Toga widgets.

## Commands

```bash
pip install briefcase        # one-time setup; app deps are declared in pyproject.toml [tool.briefcase.app.mschf]
briefcase dev                # run the Workspace Manager in dev mode

python test_microapp.py      # generates test_microapp.msf (signed code deployment + RBAC seed data)
python verify_microapp.py    # end-to-end: loads test_microapp.msf and runs it through the sandbox
python test_rbac.py          # multi-actor RBAC/signature verification (admin, support_staff, malicious_hacker)
python test_authorizer.py    # authorizer-hook enforcement (JOIN/INSERT..SELECT smuggling, PRAGMA/ATTACH — all denied)
python test_reactive.py      # reactive-redraw plumbing (on_commit notify + external-change marker, loop prevention)
python test_ledger_audit.py  # ledger replay audit (clean pass + detects out-of-band row/ledger/trigger tampering)
python test_starter.py       # starter-app authoring (signed container, trigger stamping, replay-audit clean)

python dev_tracker.py init   # dogfood: build dev_tracker.msf, a task-board micro-app tracking this project's own backlog
python dev_tracker.py list   # …also: add "title" ["detail"] | status <id> <backlog|in_progress|done> | update-app | audit | verify
```

There is no pytest/lint setup — the test scripts are standalone integration scripts run directly with `python`. They insert `src/` into `sys.path` themselves and read/write artifacts (`.msf` files, certs) in the project root. `verify_microapp.py` depends on `test_microapp.py` having produced `test_microapp.msf` (it auto-runs it if missing).

Dependencies live in `pyproject.toml` under `[tool.briefcase.app.mschf].requires` (toga, cryptography, pytz, toml, dill) — there is no requirements.txt. Briefcase's dev venv is in `.briefcase/` (generated, don't touch).

## Architecture

Code lives in `src/mschf/` (src layout, managed by Briefcase).

- **`app.py`** — the `Mschf` Toga App (Workspace Manager). On startup it: generates a Root CA (`ca.crt`/`ca.key`) and an admin identity (`admin.crt`/`admin.key`) in the project root if missing; **starts logged-out** (`Identity.logged_out()` — the admin file exists but is not auto-activated, so Open/Browse are disabled until the user authenticates); loads plugins; scans cwd + project root for `*.msf` files. You log in through the Auth Gateway plugin, which either activates an existing CA-signed identity file by CN — unlocking its **passphrase-encrypted** private key with the passphrase typed in the password field (this is how you become the seeded `admin` without the mock password minting it; the admin key's passphrase comes from `MSCHF_ADMIN_PASSPHRASE`, demo default `changeit`) — or provisions a fresh per-user identity via a mock provider (encrypting the new key with the login password). Private keys for GUI identities are stored encrypted; `admin.key` is auto-upgraded from any legacy plaintext form on startup (`_ensure_key_encrypted`, keypair preserved). The active `Identity` holds the passphrase in memory so the sandbox can sign; `set_active_identity()` enables the buttons and live-redraws all open documents. (The three test scripts still generate plaintext keys and sign directly — passphrase args default to `None`, so they're unaffected.)
- **`msf.py`** — `MSF(toga.Document)`, the document class for `.msf` files. `redraw()` checks database-level read permission for the active identity, executes the manifest's `entry_point` code via the sandbox, and wraps the returned widget with a signature-status banner ("CRYPTO ACTIVE: VERIFIED" vs tampered). Falls back to an "About" view rendered from the manifest when no entry point exists. **Reactive redraw**: each document subscribes to its storage's `on_commit` (mutating signed transactions → `app.notify_msf_commit` redraws other open documents on the same file), and `check_external_change()` (polled every ~2s by `app.on_running`) redraws when *another connection* mutated the file — `PRAGMA data_version` as the cheap gate, the ledger's non-SELECT high-water mark to ignore audit rows from signed reads (else co-open documents would redraw each other forever).
- **`storage.py`** — `MSFStorage`, the SQLite container handler. Initializes the system tables (`manifest`, `source_code`, `transactions`, `rbac_rules`, `user_roles`). The heart is `execute_signed(query, params, signature, pub_key_pem)`: verifies the payload signature (RSA PKCS#1 v1.5 / SHA-256) → verifies the cert chains to a **host-side trust anchor** (`trust.py:resolve_trust_anchors` — the host root's `ca.crt` plus any org CA in the `trusted_cas/` trust store / `MSCHF_TRUST_DIR` / `trust_dir` ctor arg; anchors re-resolved per check, never a CA shipped beside the `.msf`; fails closed when no anchors exist) → regex-parses the SQL to derive (operation, table) → enforces database-level then object-level RBAC (system tables are admin-only) → executes **under a SQLite authorizer** and appends to the `transactions` audit log. The regex checks are a coarse first gate kept for their error messages; the authorizer (`_make_authorizer` + `_rbac_snapshot`, installed around each signed execute) is the authoritative enforcement — the engine consults it at prepare time for every table/column the compiled statement touches (joins, subqueries, CTEs, views, triggers), system tables are admin-only for all operations, and `PRAGMA`/`ATTACH`/`DETACH`/vtable DDL are denied for everyone. Installing the authorizer expires SQLite's prepared-statement cache, so no signed query can reuse a statement compiled without enforcement. The connection also registers a `current_signer()` SQL function returning the verified identity string (`cert:CN=...`) of the in-flight signed transaction (NULL outside one) — container triggers use it to stamp audit/attribution columns from the cryptographic identity instead of trusting app-supplied values (see `dev_tracker.py`'s `AUDIT_TRIGGERS` for the canonical pattern, including the created-fields immutability guard; remember `recursive_triggers` is off, but a trigger's statements still fire *other* triggers). First-writer-becomes-`admin` bootstrapping is **opt-in**: it only fires when `allow_bootstrap=True`, reachable solely through the deliberate `bootstrap_admin()` authoring method — the sandbox/running path never sets it, so opening or running a `.msf` can't make you its admin. Code blobs are stored/loaded with `dill`; `get_code_signature_status()` re-verifies a blob against its signing transaction, detects tampering, **and** requires the signer to chain to the trusted CA before reporting `verified` (a valid signature from an untrusted signer is not "VERIFIED").
- **`starter.py`** — authoring for the "Getting Started" starter micro-app, created by the first-run **Create Starter App** button. The UI source lives in `STARTER_SOURCE` and is `exec`'d into a non-importable namespace before pickling so dill serializes it **by value** (the container carries its own code). `create_starter_container(dest, identity, ca_cert_path)` signs everything with the active identity (bootstrap → signed trigger DDL → seed notes → signed code blob) so the result passes `replay_audit`. First-run UX lives in `app.py`: `_update_onboarding()` drives a contextual banner (sign-in hint with the admin-passphrase tip when logged out; workspace location + starter creation when signed in and empty; hidden otherwise), the Auth Gateway prefills `admin` with the same hint, and `refresh_workspace` also scans `self.paths.data` (where the starter app is written).
- **`audit.py`** — `replay_audit(storage)` rebuilds a shadow in-memory database by replaying the signed `transactions` ledger in order and diffs it against the live tables; any divergence means the file was mutated out-of-band (a raw sqlite3 write that skipped `execute_signed`). Every ledger row's signature + CA-trust is re-verified; `datetime('now')` is overridden on the shadow connection to replay each row's recorded timestamp (so time-stamped columns/triggers reproduce); table schemas are pre-seeded from the live file but **triggers are replayed from their signed DDL** (so guards/stamping fire only where they historically existed); the ledger is copied progressively so ledger-reading statements see history as it was; `source_code` is excluded from the row diff and covered instead by folding in `get_code_signature_status`. Timestamp differences ≤2s are reported as skew (non-failing). Surfaced via `dev_tracker.py audit`.
- **`trust.py`** — host trust-anchor resolution. `resolve_trust_anchors(ca_cert_path=None, trust_dir=None)` returns the de-duplicated CA PEMs from the host `ca.crt` plus every `*.crt`/`*.pem` in the trust store (`trust_dir` arg → `MSCHF_TRUST_DIR` env → `trusted_cas/` in the host root); `is_cert_trusted(cert, anchors)` accepts a chain to *any* anchor and is False for empty anchors (callers fail closed). Must not import `storage` (storage imports it).
- **`identity.py`** — `Identity`, the single source of truth for the active user. `Identity.load(cert_path, ca_cert_path=None, trust_dir=None)` reads a cert, extracts the CN, verifies it chains to a trust anchor (fail-closed when none resolve), and locates the sibling `<stem>.key`. It bundles `cn`, `cert_path`, `key_path`, `cert_pem`, and `is_valid`. `app.py` holds one on `self.active_identity`; nothing downstream reconstructs a key filename from a CN.
- **`sandbox.py`** — `execute_micro_app(code_func, ...)` calls the unpickled callable as `code_func(toga, host_api)` and expects a Toga widget back. `HostAPI` is the only bridge exposed to micro-apps: workspace-scoped file reads, RBAC permission checks (view/field/database level), current-user info, and `execute_signed_query()` which signs on behalf of the active user using the `key_path` carried by the active `Identity` (passed in via `msf.py`).
- **`plugins/`** — `PluginManager` loads built-ins listed in `load_all()` (currently only `AuthPlugin`). Plugins subclass `plugins/base.py:BasePlugin` and may implement `extend_ui(app, outer_box)` to inject panels into the main window. The auth plugin (`plugins/auth/`) offers several protocols via an authoritative dropdown: "Existing Identity — Key Passphrase" (unlock an on-host `<CN>.crt`/`.key` by its passphrase — the default, and how you log in as `admin`), a PBKDF2 password mock, **real Google OIDC** (`providers/oidc.py` — browser authorization-code + PKCE flow, loopback redirect, ID-token RS256 verification against Google's JWKS; configured via `MSCHF_GOOGLE_CLIENT_ID`/`MSCHF_GOOGLE_CLIENT_SECRET`, and a no-op with a clear message when unset), and simulated Microsoft/passkey providers (labelled `SIMULATED`). Interactive providers (`interactive = True`) run off the UI thread via an async handler + executor. On successful IdP auth it provisions an ephemeral CA-signed X.509 cert for the identity and hot-swaps it as active; existing-identity login just activates the file.

### Signing protocol (must stay consistent)

Payload canonicalization lives in ONE place: `storage.py`'s module-level `canonical_payload(query, params, seq=None, prev_hash=None)` (bytes params base64-encoded, `json.dumps(..., sort_keys=True)`). Every signer and verifier — `execute_signed`, `get_code_signature_status`, `HostAPI.execute_signed_query`, `replay_audit`, `starter.py`, `dev_tracker.py`, and all test scripts — imports it; never re-implement it inline. The ledger is a **hash chain**: signers call `MSFStorage.get_chain_head()` → `(next_seq, prev_hash)` immediately before signing and embed both in the payload (`prev_hash` = SHA-256 of the previous row's payload+signature via `ledger_row_hash`, `''` at genesis). `execute_signed` re-derives the expected head itself under a `BEGIN IMMEDIATE` transaction — a signature over a stale head fails closed, and every signed transaction (reads included) advances the head, so sign-then-execute must be adjacent. Rows signed before chaining (NULL `seq` in `transactions`) verify under the legacy no-seq payload; the first chained row of a legacy container anchors to the last legacy row's hash. `replay_audit` flags seq gaps / prev_hash mismatches / legacy-after-chained rows as `chain_breaks` (tail truncation is inherently undetectable without an external head record).

RBAC identities are derived from the cert/key, not usernames: `cert:CN=<common_name>` for certs, `key:<sha256-prefix>` for bare public keys (see `MSFStorage._get_identity`). RBAC has four levels: `database`, `object`, `view`, `field` (field targets support `table.field`, `table.*`, `*` specificity).

## Notes

- **Versioning is semver**, maintained in three places that must move together: `pyproject.toml` (`[tool.briefcase] version`), `src/mschf/__init__.py` (`__version__`, the value the About dialog shows via `app.py:main()`), and a `CHANGELOG.md` entry. Pre-1.0: bump MINOR for features, PATCH for fixes.
- Windows is the primary target (WinForms-specific tweaks in `app.py` guarded by try/except); keep changes platform-tolerant.
- Runtime artifacts in the project root are generated, not source: `ca.crt`/`ca.key`, `admin.crt`/`admin.key`, provisioned `*.crt`/`*.key`, `*.msf` files, `settings.toml` (auto-created), `logs/`.
- `docs/ADMIN_GUIDE.md` (key generation, role assignment, signed transactions) and `docs/USER_GUIDE.md` (micro-app entry points, HostAPI usage) document the platform APIs — update them when changing those surfaces.
